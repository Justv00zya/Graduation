# Подробные требования к проекту

## 1. Общие положения

### 1.1. Назначение документа

Настоящий документ устанавливает подробные требования к информационной системе «ВузяПринт», разрабатываемой для автоматизации деятельности предприятия по ремонту и продаже оргтехники (принтеры, МФУ, сканеры, копиры, расходные материалы и комплектующие). Требования предназначены для использования при проектировании, разработке, приёмочном тестировании и сопровождении системы.

### 1.2. Область применения системы

Система охватывает следующие области деятельности организации:

- приём и ведение заявок на ремонт оргтехники от момента приёма до выдачи отремонтированного оборудования клиенту;
- учёт клиентов, поставщиков, товаров и запасных частей;
- оформление продаж техники и расходных материалов;
- учёт сотрудников, должностей и видов работ;
- формирование документов (в том числе PDF по заявке) и отчётной информации;
- управление учётными записями пользователей и разграничение доступа по ролям.

### 1.3. Пользователи и заинтересованные стороны

- **Сотрудники организации** — менеджеры, инженеры, бухгалтеры, директор; работают с системой через веб-интерфейс после входа под учётной записью с соответствующей ролью.
- **Администратор системы** — настраивает учётные записи, назначает роли, имеет полный доступ ко всем разделам.
- **Клиенты** — могут подавать заявку на ремонт через форму на главной странице без регистрации; при необходимости в системе может быть предусмотрена роль «Клиент» с ограниченным доступом после входа.
- **Внешние приложения** — мобильные или десктопные клиенты, интеграционные сервисы; взаимодействуют с системой через REST API при наличии действительного токена доступа.

### 1.4. Связь с другими документами

Требования согласуются с постановкой задачи, введением, перечнем входных данных и отчётом о тестировании проекта. Детализация реализации (структура БД, описание API, инструкция по развёртыванию) может быть приведена в отдельных разделах курсовой работы или в приложениях.

## 2. Функциональные требования

### 2.1. Аутентификация и авторизация

#### 2.1.1. Вход в систему

- Система должна обеспечивать вход пользователя по имени (логину) и паролю через веб-интерфейс на отдельной странице входа (например, «Вход» / «Login»).
- Поле «Имя пользователя» может допускать ввод логина или электронной почты в зависимости от реализации; поле «Пароль» должно отображаться в маскированном виде.
- При успешной аутентификации пользователь перенаправляется на главную страницу; в интерфейсе отображаются пункты меню и разделы в соответствии с его ролью.
- При неверном логине или пароле должно выводиться сообщение об ошибке без раскрытия информации о том, какой именно элемент (логин или пароль) неверен, с целью снижения риска перебора.
- Должна быть предусмотрена возможность выхода из системы (кнопка «Выход» / «Logout») с завершением сеанса и перенаправлением на главную страницу или страницу входа.

#### 2.1.2. Регистрация пользователей

- Должна быть реализована возможность регистрации новых учётных записей (страница «Регистрация» с полями: имя пользователя, электронная почта, пароль, подтверждение пароля и при необходимости иные поля).
- Регистрация может быть доступна только администратору (новый пользователь создаётся в разделе «Администрирование» с назначением роли) либо также открыта для самостоятельной регистрации с последующим назначением роли по умолчанию (например, «Клиент») — в зависимости от настроек проекта.
- При регистрации должны проверяться: уникальность имени пользователя и при необходимости электронной почты, соответствие пароля политике сложности (минимальная длина, при необходимости — наличие цифр, символов и т.д.), совпадение пароля и подтверждения. При нарушении требований выводятся сообщения об ошибках на русском языке.

#### 2.1.3. Восстановление пароля

- Должна поддерживаться процедура восстановления пароля (напоминание пароля): пользователь вводит логин или электронную почту, система отправляет ссылку или одноразовый код для сброса пароля (на указанный при регистрации адрес электронной почты или в рамках упрощённого механизма в режиме разработки — вывод в логи).
- Страница смены пароля по ссылке/коду должна запрашивать новый пароль и подтверждение; после успешной смены пользователь может войти с новым паролем.

#### 2.1.4. Разграничение доступа в веб-интерфейсе

- Доступ к разделам системы (Заявки, Товары, Продажи, Клиенты, Поставщики, Запчасти, Сотрудники, Отчёты, Администрирование) должен определяться ролью пользователя: в меню отображаются только те пункты, к которым у текущего пользователя есть право доступа.
- Неавторизованному пользователю доступны: главная страница (включая форму быстрой заявки), страница входа и при необходимости страница регистрации. Попытка перехода по прямой ссылке к защищённому разделу должна приводить к перенаправлению на страницу входа или на страницу «Доступ запрещён».
- Проверка прав при выполнении операций (создание, изменение, удаление записей) должна выполняться на сервере; скрытие кнопок и пунктов меню в интерфейсе является только удобством и не заменяет серверную проверку.

#### 2.1.5. Аутентификация для API

- Для программного доступа к данным система должна поддерживать аутентификацию по токену (рекомендуется JWT — JSON Web Token). Клиент получает токен путём отправки запроса на специальный метод (например, POST с логином и паролем); в последующих запросах к защищённым методам API токен передаётся в заголовке (например, «Authorization: Bearer &lt;token&gt;»).
- Срок действия токена должен быть ограничен (например, несколько часов или суток); по истечении срока клиент должен запросить новый токен (повторным входом или через механизм обновления токена, если он реализован).
- Запросы к API без токена или с недействительным/истёкшим токеном должны получать ответ с кодом 401 (Unauthorized); при недостатке прав на выполнение операции — 403 (Forbidden).

### 2.2. Заявки на ремонт

#### 2.2.1. Назначение и жизненный цикл

- Заявка на ремонт — центральная сущность системы, отражающая обращение клиента с целью ремонта оргтехники. Заявка проходит этапы от приёма до выдачи оборудования клиенту; в системе это отражается статусом заявки и заполнением полей (исполнитель, стоимость, дата выполнения, состав запчастей и работ).

#### 2.2.2. Состав данных заявки

- **Номер заявки** — уникальный идентификатор (например, автоматически формируемый при создании: порядковый номер, код с датой и т.п.); обязательное поле, только для чтения после создания (или редактируемое по правилам организации).
- **Клиент** — ссылка на справочник клиентов; обязательное поле; при выборе клиента отображаются его ФИО/наименование, телефон, адрес для контроля.
- **Модель оборудования** — текстовая строка (модель принтера, МФУ и т.д.); обязательное поле.
- **Описание состояния** — необязательное текстовое поле (внешний вид, комплектность и т.п.).
- **Описание неисправности (жалоба)** — текстовое поле с описанием проблемы со слов клиента; обязательное или рекомендуемое.
- **Исполнитель** — ссылка на справочник сотрудников (инженер, мастер); необязательное поле; назначается при постановке заявки в работу.
- **Стоимость ремонта** — денежная сумма; может заполняться при завершении работ или в процессе; необязательное поле.
- **Дата приёма заявки** — дата и при необходимости время приёма; заполняется автоматически при создании или вручную в зависимости от правил.
- **Дата выполнения** — дата (и время) завершения ремонта; необязательное поле, заполняется при переводе заявки в статус «Выполнен» или вручную.
- **Статус заявки** — перечисление: «Принят», «В работе», «Выполнен», «Отменён»; обязательное поле; определяет отображаемую информацию и доступные действия (например, назначение исполнителя при «Принят», закрытие при «В работе»).

#### 2.2.3. Позиции заявки: запчасти и работы

- К заявке должны быть привязаны **позиции по запчастям**: для каждой позиции указываются запчасть (выбор из справочника), количество. При добавлении позиции система может проверять остаток запчасти на складе и при сохранении заявки уменьшать остаток (в объёме, предусмотренном проектом).
- К заявке должны быть привязаны **выполненные виды работ**: для каждой работы указываются вид работ (выбор из справочника) и порядковый номер. Цена работы может браться из справочника видов работ; итоговая стоимость заявки может рассчитываться по работам и запчастям или вводиться вручную.

#### 2.2.4. Операции над заявками

- **Создание заявки** — заполнение перечисленных полей, при необходимости добавление позиций запчастей и работ; сохранение с присвоением номера и статуса «Принят». Доступно пользователям с ролями, имеющими право на создание заявок (например, менеджер, администратор).
- **Просмотр заявки** — отображение всех полей заявки, списка использованных запчастей и выполненных работ. Должна быть возможность открыть заявку из списка (по номеру или из карточки клиента).
- **Редактирование заявки** — изменение полей и состава позиций в соответствии с правами роли и бизнес-правилами (например, запрет редактирования после перевода в «Выполнен» или разрешение только для определённых ролей).
- **Удаление заявки** — возможность удаления заявки (как правило, только для администратора или только заявок в статусе «Отменён»); при удалении должны учитываться связи (например, каскадное удаление позиций заявки по запчастям и работам).

#### 2.2.5. Список заявок и фильтрация

- На странице «Заявки» должен отображаться список заявок (таблица или карточки) с основными полями: номер, дата, клиент, модель оборудования, статус, исполнитель, стоимость. Должна быть поддержка сортировки (по дате, статусу, номеру) и при необходимости постраничная навигация.
- Должна быть возможность фильтрации списка: по статусу, по периоду дат, по клиенту, по исполнителю — в объёме, предусмотренном реализацией.

#### 2.2.6. Печатная форма и PDF

- Должна быть реализована возможность формирования печатной формы или PDF-документа по заявке (кнопка «Печать» / «Сформировать PDF» на странице просмотра заявки). В документе должны отображаться: номер заявки, дата, данные клиента, модель оборудования, описание неисправности, статус, исполнитель, стоимость, состав запчастей и работ. Документ предназначен для выдачи клиенту или для внутреннего учёта.

### 2.3. Клиенты

#### 2.3.1. Назначение справочника

- Справочник клиентов используется при оформлении заявок на ремонт и продаж; по клиенту можно получать историю заявок и продаж.

#### 2.3.2. Состав данных

- **ФИО или наименование организации** — текстовая строка; обязательное поле; допускается ввод как физического лица (ФИО), так и наименования организации.
- **Адрес** — необязательное текстовое поле (адрес доставки, юридический адрес и т.п.).
- **Телефон** — необязательное, но рекомендуемое поле для связи; при приёме быстрой заявки по телефону система может искать существующего клиента по нормализованному номеру (цифры) и при совпадении привязывать заявку к нему, иначе создавать нового клиента.

#### 2.3.3. Операции

- **Просмотр списка клиентов** — таблица или список с полями: ФИО/наименование, телефон, адрес; возможность перехода к карточке клиента и при необходимости к списку его заявок и продаж.
- **Создание клиента** — форма с полями ФИО/наименование, адрес, телефон; проверка обязательных полей; сохранение с присвоением идентификатора.
- **Редактирование клиента** — изменение полей существующей записи; при наличии связанных заявок или продаж удаление клиента может быть запрещено или ограничено (например, только снятие привязки или каскад по правилам БД).
- **Удаление клиента** — при отсутствии связей или в соответствии с правилами целостности (например, запрет удаления при наличии заявок; при необходимости — «мягкое» удаление или архив).

#### 2.3.4. Связь с быстрой заявкой

- При отправке формы быстрой заявки с главной страницы система должна по указанному телефону (после нормализации) искать существующего клиента; при отсутствии — создавать новую запись клиента с полями «ФИО/наименование» и «Телефон» из формы; заявка связывается с найденным или созданным клиентом.

### 2.4. Товары

#### 2.4.1. Назначение справочника

- Справочник товаров используется для учёта техники и расходных материалов, продаваемых организацией; товары участвуют в позициях продаж, по ним ведётся остаток на складе.

#### 2.4.2. Состав данных

- **Код товара** — уникальный идентификатор (артикул, штрихкод и т.п.); обязательное поле.
- **Поставщик** — ссылка на справочник поставщиков; обязательное поле при создании/редактировании.
- **Наименование** — текстовая строка; обязательное поле.
- **Модель** — необязательное поле (модель техники, вариант комплектации).
- **Цена** — денежная величина; обязательное поле; должна быть неотрицательной.
- **Количество на складе** — целое неотрицательное число; при оформлении продажи количество по позициям списывается с остатка товара (контроль неотрицательного остатка — в соответствии с реализацией).

#### 2.4.3. Операции

- **Просмотр списка товаров** — отображение кода, наименования, модели, поставщика, цены, остатка; возможность перехода к редактированию и при необходимости к истории продаж по товару.
- **Создание и редактирование товара** — форма с полями кода, поставщика, наименования, модели, цены, количества; валидация типов и обязательных полей.
- **Удаление товара** — при отсутствии позиций продаж или по правилам (например, запрет удаления при наличии продаж; каскад или ограничение в зависимости от требований).

---

### 2.5. Запчасти

#### 2.5.1. Назначение справочника

- Справочник запчастей используется для учёта запасных частей и расходников, используемых при ремонте; запчасти привязываются к заявкам в виде позиций с указанием количества, остаток на складе при этом уменьшается (в объёме, предусмотренном проектом).

#### 2.5.2. Состав данных

- **Код запчасти** — уникальный идентификатор; обязательное поле.
- **Поставщик** — ссылка на справочник поставщиков; обязательное поле.
- **Наименование** — текстовая строка; обязательное поле.
- **Цена** — денежная величина; обязательное поле; неотрицательная.
- **Количество на складе** — целое неотрицательное число; при добавлении позиции запчасти к заявке остаток уменьшается; при снятии позиции или отмене заявки — может восстанавливаться в зависимости от реализованной логики.

#### 2.5.3. Операции

- **Просмотр списка запчастей** — код, наименование, поставщик, цена, остаток; переход к редактированию и при необходимости к списку заявок, в которых использована запчасть.
- **Создание и редактирование запчасти** — форма с полями кода, поставщика, наименования, цены, количества; валидация.
- **Удаление запчасти** — при отсутствии использования в позициях заявок или по правилам целостности.
- Доступ к разделу запчастей и к операциям создания/редактирования/удаления ограничивается ролями (например, инженер, директор, администратор).

---

### 2.6. Поставщики

#### 2.6.1. Назначение справочника

- Справочник поставщиков используется при ведении товаров и запчастей (у каждого товара и запчасти указывается поставщик); при необходимости по поставщику можно формировать отчёты или выборки.

#### 2.6.2. Состав данных

- **Наименование** — полное наименование организации; обязательное поле.
- **Адрес** — необязательное поле (юридический или фактический адрес).
- **ИНН** — необязательное поле; при вводе может проверяться формат (цифры, длина).
- **Расчётный счёт** — необязательное поле (банковские реквизиты).
- **Телефон** — необязательное поле для связи.

#### 2.6.3. Операции

- **Просмотр списка поставщиков** — наименование, ИНН, телефон, адрес; переход к карточке и при необходимости к списку товаров и запчастей данного поставщика.
- **Создание, редактирование и удаление поставщика** — в соответствии с правами роли; при удалении необходимо учитывать наличие товаров и запчастей, привязанных к поставщику (запрет удаления или переназначение в зависимости от правил).

### 2.7. Продажи

#### 2.7.1. Назначение

- Продажа отражает факт реализации товаров клиенту (или без указания клиента); фиксируются дата, состав позиций (товар, количество, цена за единицу, сумма по позиции) и итоговая сумма.

#### 2.7.2. Состав данных продажи

- **Номер продажи** — уникальный идентификатор (формируется автоматически или вручную).
- **Клиент** — ссылка на справочник клиентов; необязательное поле (продажа может быть без привязки к клиенту).
- **Дата продажи** — дата и при необходимости время; обязательное поле; по умолчанию — текущая дата.
- **Итоговая сумма** — денежная величина; может рассчитываться автоматически как сумма по всем позициям или вводиться вручную с контролем соответствия сумме позиций.

#### 2.7.3. Позиции продажи

- Каждая позиция содержит: ссылку на товар, количество, цену за единицу (может подставляться из справочника товаров с возможностью изменения), сумму по позиции (количество × цена за единицу). Итоговая сумма продажи равна сумме полей «сумма по позиции» по всем позициям.

#### 2.7.4. Операции

- **Создание продажи** — ввод номера (или автоматическое присвоение), выбор клиента (опционально), даты; добавление позиций (выбор товара, ввод количества и при необходимости цены); расчёт итога; сохранение. При сохранении остатки товаров на складе уменьшаются на количество по каждой позиции (с проверкой неотрицательного остатка в объёме, предусмотренном проектом).
- **Просмотр списка продаж** — таблица с номером, датой, клиентом, итоговой суммой; сортировка и при необходимости фильтрация по дате, клиенту.
- **Просмотр деталей продажи** — открытие карточки продажи с полным составом позиций (товар, количество, цена, сумма) и итогом.

---

### 2.8. Сотрудники и должности

#### 2.8.1. Справочник должностей

- **Наименование должности** — текстовая строка (например: Менеджер, Бухгалтер, Инженер, Директор); обязательное поле.
- **Оклад** — денежная величина; обязательное поле; неотрицательное.
- Операции: просмотр списка должностей, создание, редактирование, удаление (с учётом наличия сотрудников с данной должностью).

#### 2.8.2. Справочник сотрудников

- **Табельный номер** — уникальный идентификатор сотрудника в организации; обязательное поле.
- **Должность** — ссылка на справочник должностей; обязательное поле.
- **Фамилия, имя, отчество** — обязательные поля (отчество может быть необязательным).
- **Дата рождения** — обязательное поле; тип «дата».
- **ИНН** — необязательное поле; при вводе может проверяться формат.
- **Адрес** — необязательное поле.
- **Дата приёма на работу** — обязательное поле; тип «дата».
- Сотрудник может быть назначен **исполнителем** по заявке на ремонт (поле «Исполнитель» в заявке).

#### 2.8.3. Операции

- **Просмотр списка сотрудников** — табельный номер, ФИО, должность, дата приёма; переход к карточке сотрудника с полными данными и при необходимости к списку заявок, где он указан исполнителем.
- **Создание, редактирование и удаление сотрудника** — доступно пользователям с соответствующими ролями (бухгалтер, директор, администратор). При удалении сотрудника заявки, где он указан исполнителем, могут сохранять ссылку с обнулением (исполнитель не назначен) или по правилам целостности.

---

### 2.9. Виды работ

#### 2.9.1. Назначение

- Справочник видов работ определяет перечень услуг (диагностика, ремонт, заправка картриджа, чистка и т.д.) и их цену; виды работ привязываются к заявкам при учёте выполненных операций.

#### 2.9.2. Состав данных

- **Наименование вида работ** — текстовая строка; обязательное поле.
- **Цена** — денежная величина; обязательное поле; неотрицательная.

#### 2.9.3. Операции

- Просмотр списка видов работ, создание, редактирование, удаление (с учётом использования в заявках). При добавлении «работы» к заявке пользователь выбирает вид работ из справочника; порядковый номер и при необходимости цена подставляются автоматически или вводятся вручную.

---

### 2.10. Быстрая заявка на главной странице

#### 2.10.1. Доступность

- Форма быстрой заявки размещается на главной странице сайта и доступна всем посетителям без регистрации и входа в систему.

#### 2.10.2. Поля формы

- **ФИО или название организации** — обязательное поле; используется при создании или поиске клиента.
- **Телефон для связи** — обязательное поле; используется для поиска клиента по нормализованному номеру (только цифры) и при создании нового клиента при отсутствии совпадений.
- **Модель принтера (оборудования)** — обязательное поле; может быть выпадающий список типовых моделей с вариантом «Другое» и полем ввода произвольного текста.
- **Описание проблемы (неисправности)** — обязательное поле; может быть выпадающий список типовых неисправностей с вариантом «Другое» и полем для произвольного описания.

#### 2.10.3. Обработка после отправки

- Валидация обязательных полей на стороне клиента и сервера; при ошибках — вывод сообщений под полями или в блоке «Ошибки».
- По телефону (нормализованному) выполняется поиск клиента в справочнике; при отсутствии — создаётся новая запись клиента (ФИО/наименование, телефон).
- Создаётся заявка на ремонт: клиент — найденный или созданный; модель оборудования и описание неисправности — из формы; статус «Принят»; дата приёма — текущая; номер заявки формируется автоматически (например, префикс «WEB-» и дата-время).
- По желанию формируется PDF-документ по заявке и сохраняется на сервере или отдаётся пользователю; в интерфейсе выводится сообщение об успешной отправке заявки и при необходимости путь или имя сформированного файла.
- После успешной отправки форма может быть сброшена для повторной подачи заявки.

---

### 2.11. Отчётность

#### 2.11.1. Раздел отчётов

- В системе должен быть раздел «Отчёты», доступный пользователям с ролями бухгалтер, директор, администратор (и при необходимости иным ролям по решению проекта). В разделе размещаются ссылки или формы для формирования предопределённых отчётов.

#### 2.11.2. Содержание отчётов

- Отчёты формируются на основе данных, хранящихся в системе: заявки (по периодам, статусам, исполнителям), продажи (по периодам, клиентам, товарам), остатки товаров и запчастей, движение по складу, данные по сотрудникам и т.п. Конкретный перечень отчётов (список заявок, список продаж, отчёт по остаткам, свод по заявкам за период и т.д.) определяется в рамках реализации. Отчёты должны обеспечивать достоверную информацию для бухгалтерского учёта и принятия управленческих решений.

---

### 2.12. Администрирование

#### 2.12.1. Доступ

- Раздел «Администрирование» доступен только пользователям с ролью «Администратор системы». В меню пункт отображается только при наличии этой роли.

#### 2.12.2. Функции раздела

- **Управление учётными записями** — просмотр списка пользователей (логин, email, роли), создание нового пользователя (логин, email, пароль, выбор ролей), редактирование (изменение email, пароля, назначение и снятие ролей), при необходимости блокировка или удаление учётной записи.
- **Назначение ролей** — каждой учётной записи может быть назначена одна или несколько ролей из фиксированного набора: администратор, менеджер, инженер, бухгалтер, директор, клиент. От назначенных ролей зависит состав меню и права на операции в разделах и в API.

## 3. Требования к разграничению доступа по ролям

### 3.1. Сводная таблица доступа к разделам

| Главная           | да            | да       | да      | да        | да       | да     |
| Заявки            | полный доступ | полный   | редакт. | нет       | полный   | нет    |
| Товары            | полный доступ | полный   | нет     | нет       | полный   | нет    |

| Клиенты           | полный доступ | полный   | нет     | нет       | полный   | нет    |

*Полный доступ* — просмотр, создание, редактирование, удаление (с учётом бизнес-правил). *Редакт.* — редактирование заявок (например, без удаления). *Просмотр* — только просмотр списков и карточек.

### 3.2. Детализация по ролям

- **Администратор системы** — полный доступ ко всем разделам; регистрация и редактирование пользователей, назначение и снятие ролей; удаление заявок, клиентов, товаров, запчастей, поставщиков, продаж, сотрудников в соответствии с реализованными правилами целостности и бизнес-логикой.
- **Менеджер** — заявки (создание, просмотр, редактирование, при необходимости удаление в рамках правил), товары, продажи (оформление и просмотр), клиенты, поставщики (полный цикл); нет доступа к запчастям, сотрудникам, отчётам, администрированию.
- **Инженер** — заявки (просмотр и редактирование, в том числе назначение себя исполнителем, добавление запчастей и работ); запчасти (просмотр и при необходимости редактирование остатков/позиций); нет доступа к товарам, продажам, клиентам, поставщикам, сотрудникам, отчётам, администрированию.
- **Бухгалтер** — сотрудники (полный доступ к справочнику), отчёты (формирование и просмотр); продажи (просмотр для учёта); нет доступа к заявкам, товарам, клиентам, поставщикам, запчастям, администрированию.
- **Директор** — просмотр и при необходимости редактирование по всем разделам, кроме администрирования; полный доступ к заявкам, товарам, продажам, клиентам, поставщикам, запчастям, сотрудникам, отчётам.
- **Клиент** — после входа может иметь ограниченный доступ (например, просмотр своих заявок — если реализован личный кабинет); без входа — только главная страница и форма быстрой заявки.

Конкретный перечень операций (создание, изменение, удаление, просмотр) для каждой роли уточняется в реализации и должен быть описан в документации и при необходимости в отчёте о тестировании.

## 4. Требования к данным и целостности

### 4.1. Хранение данных

- Все перечисленные сущности (заявки, позиции заявок по запчастям и работам, клиенты, поставщики, товары, запчасти, должности, сотрудники, виды работ, продажи, позиции продаж, учётные записи пользователей и роли) хранятся в реляционной SQL-базе данных. Идентификация записей осуществляется по первичному ключу (например, целочисленный Id или GUID); для бизнес-идентификации используются номера заявок, продаж, коды товаров и запчастей, табельные номера и т.д.

### 4.2. Связи между сущностями

- **Заявка** — связь «многие к одному» с клиентом (обязательно), «многие к одному» с сотрудником (исполнитель, необязательно). **Позиции заявки по запчастям** — связь с заявкой (каскадное удаление при удалении заявки) и с запчастью (ограничение при удалении запчасти). **Работы по заявке** — связь с заявкой (каскад) и с видом работ (ограничение при удалении вида работ).
- **Товар, запчасть** — связь «многие к одному» с поставщиком (при удалении поставщика — ограничение или переназначение по правилам).
- **Продажа** — связь «многие к одному» с клиентом (необязательно). **Позиция продажи** — связь с продажей (каскад) и с товаром (ограничение при удалении товара).
- **Сотрудник** — связь «многие к одному» с должностью (при удалении должности — ограничение при наличии сотрудников с этой должностью).

### 4.3. Валидация и сообщения об ошибках

- Обязательные поля (номер заявки, клиент в заявке, наименование товара/запчасти, имя пользователя и пароль при регистрации, табельный номер и ФИО сотрудника и т.п.) проверяются при вводе в формах и при сохранении на сервере. Числовые поля (цена, количество, оклад) должны быть неотрицательными где это предусмотрено; даты — в допустимом диапазоне. При некорректных данных пользователю выводятся понятные сообщения на русском языке (под полем или в блоке валидации), без раскрытия внутренней структуры БД или кода.

### 4.4. Остатки на складе

- При добавлении позиции запчасти к заявке и при оформлении продажи с позициями товаров остатки на складе должны уменьшаться. В объёме, предусмотренном проектом, должна выполняться проверка неотрицательного остатка (запрет операции при недостатке или предупреждение с возможностью сохранения — по правилам организации). При отмене заявки или удалении позиции остатки могут восстанавливаться в соответствии с реализованной логикой.

## 5. Требования к программному интерфейсу (API)

### 5.1. Назначение

- REST API предназначен для взаимодействия внешних клиентов (мобильные приложения, десктоп-клиенты, интеграционные сервисы) с данными системы без использования веб-интерфейса. Все операции с данными, доступные через API, должны проверять аутентификацию и права пользователя (роль).

### 5.2. Протокол и формат

- Взаимодействие осуществляется по протоколу HTTP/HTTPS. Рекомендуется использование JSON для тела запросов и ответов. Кодировка — UTF-8. Методы HTTP: GET (получение ресурса или списка), POST (создание, а также аутентификация), PUT или PATCH (обновление), DELETE (удаление). Коды ответов: 200 (OK), 201 (Created), 400 (Bad Request — ошибка валидации), 401 (Unauthorized — не передан или недействителен токен), 403 (Forbidden — недостаточно прав), 404 (Not Found), 500 (Internal Server Error — внутренняя ошибка сервера).

### 5.3. Аутентификация

- Для защищённых методов в заголовке запроса должен передаваться JWT-токен: «Authorization: Bearer &lt;token&gt;». Токен выдаётся методом аутентификации (например, POST /api/auth/login с телом { "userName": "...", "password": "..." }); в ответе возвращается объект с полем token (и при необходимости сроком действия). Запросы без токена или с недействительным токеном возвращают 401; при недостатке прав — 403.

### 5.4. Ресурсы и методы

- **Аутентификация**: POST /api/auth/login (получение токена); POST /api/auth/register (регистрация — при необходимости только для администратора).
- **Заявки**: GET /api/orders (список), GET /api/orders/{id} (одна заявка), POST /api/orders (создание), PUT /api/orders/{id} (обновление), DELETE /api/orders/{id} (удаление — по правам).
- **Клиенты**: GET /api/clients, GET /api/clients/{id}, POST /api/clients, PUT /api/clients/{id}, DELETE /api/clients/{id}.
- **Товары**: GET /api/products, GET /api/products/{id}, POST /api/products, PUT /api/products/{id}, DELETE /api/products/{id}.
- **Запчасти**: GET /api/parts, GET /api/parts/{id} (и при необходимости POST, PUT, DELETE в зависимости от прав).
- **Сотрудники**: GET /api/employees, GET /api/employees/{id} (создание/редактирование через API может не реализовываться или ограничиваться).
- **Продажи**: GET /api/sales, GET /api/sales/{id} (и при необходимости POST для создания продаж).
- На каждый метод действуют ограничения по ролям в соответствии с разделом 3; при обращении от имени пользователя без прав возвращается 403.

### 5.5. Документирование

- API должен быть описан в формате OpenAPI (Swagger): перечень ресурсов, методов, параметров запроса и тела, структуры ответов, коды ошибок, требования к заголовку Authorization. Интерактивная документация (Swagger UI) должна быть доступна по отдельному адресу (например, /swagger) в режиме разработки или по настройке в production.

## 6. Нефункциональные требования

### 6.1. Безопасность

- **Хранение паролей**: пароли пользователей не должны храниться в открытом виде; необходимо использование криптографически стойкого хеширования (например, алгоритмы, встроенные в ASP.NET Core Identity).
- **Сеанс**: сеанс авторизованного пользователя в веб-интерфейсе должен быть привязан к учётной записи и при выходе или истечении срока действия — завершаться; при необходимости настраивается таймаут неактивности.
- **Проверка прав на сервере**: каждая операция изменения или удаления данных, а также выдача конфиденциальных данных должны проверяться на сервере по роли/правам пользователя. Скрытие кнопок и пунктов меню в интерфейсе не является достаточной мерой; запрос, выполненный в обход интерфейса (например, прямая отправка запроса к API), должен отклоняться с кодом 403 при недостатке прав.
- **Защита от типовых уязвимостей**: входные данные должны проверяться на допустимость; необходимо предотвращение инъекций (SQL, иные), кросс-сайтового скриптинга (XSS) и подделки межсайтовых запросов (CSRF) в объёме, предусмотренном используемым фреймворком (ASP.NET Core и др.).

### 6.2. Удобство использования (usability)

- Интерфейс должен быть понятным для пользователей с разным уровнем подготовки: единообразная навигация по разделам через главное меню, явные названия кнопок («Сохранить», «Отмена», «Удалить», «Добавить позицию» и т.д.) и подписи полей на русском языке. Сообщения об ошибках и подсказки — на русском языке.
- Формы ввода должны по возможности проверять данные на стороне клиента (обязательные поля, формат, диапазоны) до отправки на сервер; на сервере выполняется повторная проверка. При ошибке валидации указываются конкретные поля и причины (например, «Заполните поле "Телефон"», «Введите корректную дату»).
- Списки с большим количеством записей должны поддерживать постраничный вывод или подгрузку для удобства просмотра и снижения нагрузки.

### 6.3. Совместимость и развёртывание

- Веб-интерфейс должен корректно отображаться и функционировать в современных браузерах с поддержкой используемых технологий (например, последние версии Chrome, Firefox, Edge, Safari; при необходимости указывается перечень поддерживаемых браузеров).
- Система должна развёртываться с использованием реляционной СУБД; в качестве основной рекомендуется PostgreSQL; допускается поддержка альтернативы (например, SQLite) для упрощённой установки или разработки. Требования к версии СУБД, к платформе (.NET 8, ОС сервера) и шаги установки должны быть зафиксированы в документации по развёртыванию.

### 6.4. Надёжность и сопровождение

- При возникновении ошибок во время выполнения операций пользователю должно выводиться информативное сообщение (например, «Не удалось сохранить заявку. Проверьте введённые данные» или «Сервер временно недоступен»). Детали внутренних исключений (стек вызовов, параметры запроса) не должны отображаться конечному пользователю в production во избежание раскрытия структуры системы и конфиденциальных данных; при этом информация может записываться в журнал для анализа разработчиком.
- Исходный код и структура проекта должны обеспечивать возможность сопровождения и доработки: логичное разделение на слои (представление, бизнес-логика, доступ к данным), использование общих компонентов и соглашений по именованию, комментарии и описание ключевых решений в объёме, необходимом для курсового проекта.

### 6.5. Производительность (в рамках курсового проекта)

- Требования к времени отклика и количеству одновременных пользователей определяются рамками курсового проекта и типовой конфигурацией рабочей станции или учебного сервера. Желательно обеспечить приемлемую скорость загрузки страниц и выполнения операций при типичном объёме данных (сотни заявок, клиентов, товаров, запчастей).

## 7. Ограничения и допущения

- Перечень ролей и детальный список прав могут быть уточнены в ходе реализации в пределах описанной модели (администратор, менеджер, инженер, бухгалтер, директор, клиент).
- Регистрация новых пользователей может быть разрешена только через раздел администрирования (создание учётной записи администратором с назначением роли); страница самостоятельной регистрации может быть отключена или использоваться только для роли «Клиент».
- Восстановление пароля может быть реализовано в упрощённом виде (например, в режиме разработки ссылка или код выводится в журнал сервера вместо отправки письма).
- Отчёты реализуются в виде набора предопределённых форм (списки заявок, продаж, остатки, своды по периодам и т.п.) без обязательной поддержки произвольных пользовательских отчётов и конструктора отчётов.
- Требования к высокой доступности, резервному копированию, масштабированию и нагрузочному тестированию выходят за рамки базового курсового проекта и при необходимости формулируются отдельно.
