3. Технологическая часть (продолжение)

3.3. Защитное программирование

В программном коде веб-приложения OrgTechRepair предусмотрены меры против неожиданного прерывания работы сервиса и потери целостности данных при сбоях вспомогательных операций (файловая система, формирование PDF), а также стандартные проверки входных данных и наличия сущностей в базе перед обновлением и удалением. Наряду с атрибутами авторизации [Authorize] и проверками на null после выборок из PostgreSQL через Entity Framework Core, в критических местах используются блоки try…catch с записью предупреждений в журнал ILogger, чтобы основной сценарий (например, сохранение заявки в БД) завершался штатно даже при отказе побочного шага.

Аналогичный приём применяется при повторной генерации PDF в архив после изменения заявки (TryArchiveOrderPdfAsync): исключение при записи файла не отменяет успешное сохранение полей заказа в базе. При удалении товара вспомогательное удаление файла изображения с диска обёрнуто в try…catch с уровнем Warning в логе, чтобы сбой файловой системы не блокировал удаление записи из таблицы Products.
